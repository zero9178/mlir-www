34 /// Defines a parallelization strategy. Any independent loop is a candidate
35 /// for parallelization. The loop is made parallel if (1) allowed by the
36 /// strategy (e.g., AnyStorageOuterLoop considers either a dense or sparse

95 /// Defines a strategy for implementing sparse-to-sparse conversion.
96 /// `kAuto` leaves it up to the compiler to automatically determine
97 /// the method used. `kViaCOO` converts the source tensor to COO and
98 /// then converts the COO to the target format. `kDirect` converts
99 /// directly via the algorithm in <https://arxiv.org/abs/2001.02609>;
100 /// however, beware that there are many formats not supported by this
101 /// conversion method.
102 ///
103 /// The presence of the `kAuto` option violates our usual goal of keeping
104 /// policy completely separated from mechanism. The reason it exists is
105 /// because (at present) this strategy can only be specified on a per-file
106 /// basis. To see why this is a problem, note that `kDirect` cannot
107 /// support certain conversions; so if there is no `kAuto` setting,
108 /// then whenever a file contains a single non-`kDirect`-able conversion
109 /// the user would be forced to use `kViaCOO` for all conversions in
110 /// that file! In the future, instead of using this enum as a `Pass`
111 /// option, we could instead move it to being an attribute on the
112 /// conversion op; at which point `kAuto` would no longer be necessary.
113 enum class SparseToSparseConversionStrategy { kAuto, kViaCOO, kDirect };
